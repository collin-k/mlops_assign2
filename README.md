# MLOps Assignment 2 - Feature Store

A reproducible, end-to-end ML pipeline that predicts an athlete's `total_lift`
(sum of deadlift, clean-and-jerk, snatch, and back squat) from the
`athletes.csv` dataset. The project integrates feature management, experiment 
tracking, versioning, and model evaluation.

## Stack

| Concern | Tool | Reason |
|---|---|---|
| Feature store | **Feast** | Open-source, runs fully local (file offline store + SQLite registry/online store), and lets us define, version, store, and retrieve features with point-in-time-correct joins. |
| Experiment tracking | **MLflow** | Logs parameters, feature version, metrics, and model artifacts for every run and makes the 4 experiments easy to compare. |
| Model | **XGBoost** | Single algorithm used across all experiments (no AutoML), strong tabular-regression baseline. |

## Repository structure

```
data/athletes.csv          raw dataset
feature_repo/
  feature_store.yaml        Feast config
  definitions.py            entity + two versioned feature views (v1, v2)
  data/                     generated feature parquet + Feast registry/online store
notebooks/eda.ipynb         exploratory analysis + visualizations
reports/
  experiment_results.csv    metrics for the 4 runs
  experiment_summary.md     comparison + discussion
  figures/                  prediction plots + comparison chart
pipeline.py                 the pipeline logic (ingest -> ... -> 4 experiments)
config.py                   all tunable values (paths, seed, features, hyperparams)
requirements.txt            pinned dependencies
```

Everything under `feature_repo/data/` (feature parquet, `registry.db`,
`online_store.db`) is generated by running the pipeline and is git-ignored.

## Setup

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Place the `athletes.csv` in `data/` before running.

## Reproduce the results

```bash
python pipeline.py
```

This runs the whole workflow and writes `reports/experiment_results.csv`,
figures, and MLflow runs. Browse the tracked experiments with:

```bash
mlflow ui --backend-store-uri ./mlruns
```

To also run the EDA notebook:

```bash
jupyter lab notebooks/eda.ipynb
```

## Pipeline stages (`pipeline.py`)

1. **Ingest + preprocess** - load the CSV; drop blanks / "decline to answer",
   constrain physical measures to plausible ranges (removing extreme outliers),
   require positive lifts, and build the `total_lift` target.
2. **Feature engineering** - write two versioned feature tables to parquet.
3. **Feast apply** - register the entity and both feature views.
4. **Retrieve + train** - pull each feature version from Feast with
   `get_historical_features`, train XGBoost, and log everything to MLflow.

## Feature versions (Feast feature views)

Two feature views are registered in the same store, giving managed feature
*versions*.

| Version | View | Features |
|---|---|---|
| v1 (baseline) | `athlete_features_v1` | `age`, `height`, `weight`, `gender_male` |
| v2 (engineered) | `athlete_features_v2` | v1 + `bmi`, `weight_to_height`, `age_bucket`, `is_experienced` |

**Difference:** v2 adds body-composition ratios (`bmi`, `weight_to_height`), a
bucketed age (`age_bucket`), and an experience flag (`is_experienced`) on top of
the baseline physical attributes.

## Experiments

One algorithm (XGBoost) x 2 feature versions x 2 hyperparameter configs = 4 runs.

| Config | n_estimators | max_depth | learning_rate | subsample | colsample_bytree |
|---|---|---|---|---|---|
| shallow | 100 | 3 | 0.10 | 0.8 | 0.8 |
| deep | 400 | 6 | 0.05 | 0.9 | 0.9 |

All tunable values live in [config.py](config.py); `pipeline.py` contains only
logic. Metrics for the four runs are written to
[reports/experiment_results.csv](reports/experiment_results.csv), with the
comparison chart at `reports/figures/experiment_comparison.png`.

## Assumptions

- Regression target `total_lift = deadlift + candj + snatch + backsq`; the
  component columns are excluded from every feature set to prevent leakage.
- 80/20 train/test split, `random_state=42` for reproducibility.
- Feast runs entirely locally - no cloud account required.
- After cleaning, ~47.1k of the ~423k raw rows remain.
